# TINA

**TINA** is a hybrid grapheme-to-pronunciation system for **Brazilian Portuguese**. Rather than treating pronunciation as a flat character-to-phone mapping, the model predicts a **structured, archiphonemic representation** with explicit **syllabification** and **lexical stress**, then expands that representation into **surface-oriented dialectal variants** through a rule-based phonological module.

The project is designed for research on Brazilian Portuguese pronunciation structure, especially in settings where it is important to preserve distinctions that are often flattened away in conventional G2P pipelines.

## What the system does

Given orthographic input, TINA produces:

1. a **base transcription** that is:
   - syllabified,
   - stress-marked,
   - partially archiphonemic;
2. a **set of expanded variants** generated by explicit phonological rules.

This makes the system useful both as:

- a **working transcription tool**, and
- a **research model** for structured pronunciation generalization on lexical items and pseudowords.

## Core idea

TINA separates the pronunciation problem into two stages:

- **Neural stage**: predicts a structured pronunciation representation from orthography.
- **Symbolic stage**: expands that representation into controlled surface variants.

This division is intentional. The neural model handles stable regularities such as syllable count, syllable sequencing, segmental structure, and stress placement. The rule layer handles dialect-sensitive or context-sensitive realization, especially in domains where Brazilian Portuguese displays substantial variation.

## Architecture overview

The implemented model is a shared encoder plus three coordinated decoders.

### 1. Encoder

The encoder consumes character-level input and builds contextualized representations.

Pipeline:

- character embedding layer
- 1D convolution (`kernel_size=3`) + batch normalization + ReLU
- residual addition with the original embeddings
- 2-layer bidirectional GRU
- layer normalization
- projection MLP
- Transformer encoder layer
- auxiliary syllable-count prediction head

The encoder outputs:

- contextualized token representations,
- a compressed hidden state used to initialize decoding,
- a predicted syllable count.

### 2. Syllable decoder

The first decoder predicts **whole syllable tokens**, not just boundary positions.

Key properties:

- GRU decoder with Luong-style attention over encoder outputs
- special sequence markers: `<bow>`, `<eow>`, `<pad_syl>`
- token branch for next-syllable prediction
- boundary branch for auxiliary boundary/count supervision
- optional forcing of `<eow>` using the expected or predicted syllable count

### 3. Phoneme decoder

For each predicted syllable, a second decoder generates the internal phoneme sequence.

Key properties:

- GRU decoder initialized from the syllable hidden state
- Luong-style attention over encoder outputs
- autoregressive decoding from `<bop>` to `<eop>`
- residual refinement before projection to phoneme logits
- hidden states are mean-pooled per syllable and passed to the stress module as phoneme-derived features

### 4. Stress decoder

The final decoder predicts stress over the syllable sequence.

Key properties:

- positional embeddings over syllable positions
- bidirectional GRU over syllable states
- Transformer encoder layer over syllable-level context
- phoneme-feature projection and fusion
- another attention pass over the encoder states
- binary stress classification per syllable

## Training objectives

Training is multi-objective. The implementation combines:

- **syllable loss**: cross-entropy with label smoothing
- **phoneme loss**: cross-entropy with label smoothing
- **stress loss**: focal loss with class weighting
- **alignment/count loss**: MSE-style supervision on syllable count estimates
- **boundary loss**: auxiliary end-of-word/boundary supervision

Current weights in `training.py`:

- `syl`: `0.30`
- `stress`: `0.30`
- `phon`: `0.20`
- `align`: `0.20`
- boundary auxiliary weight: `0.10`

Teacher forcing decays exponentially during training:

```python
teacher_forcing_ratio = math.exp(-epoch / 30.0)
```

## Rule-based phonological expansion

After neural inference, the base transcription is expanded into surface-oriented variants by `phonological_rules.py`.

Implemented processes include:

- consonant-coda epenthesis with canonical BP epenthetic `/i/`
- `/s/`-cluster epenthesis in onset-like and coda-like environments
- expansion of abstract vowel markers such as `I` and `U`
- context-sensitive coronal alternations after `t` and `d`
- stress-conditioned vowel harmony
- dialect-profile expansion for ambiguous symbols such as `H`, `T`, `D`, `S`, `R`, `L`, `E`, and `O`
- special rhotic handling, including optional final-`R` deletion and `RS`-cluster reduction in eligible environments

### Dialect profiles currently implemented

- `Nortista_Nordestino`
- `Sulista_Sudestino`
- `Carioca`

The rule engine returns a **deduplicated set of variants** rather than a single canonical surface form.

## Data format

The training lexicon is expected at:

```text
data/data.txt
```

Each line should contain:

```text
<lemma_or_word><whitespace><phonological_transcription>
```

Example shape:

```text
palavra /pa.ˈla.vɾa/
```

Important conventions used by the code:

- syllables are separated with `.`
- stress is marked with prefixed `ˈ` on the stressed syllable
- the transcription may contain **archiphonemic symbols** rather than fully resolved surface phones
- the training loader strips outer slashes and derives the expected syllable count from the number of syllable separators

## Training

Run:

```bash
python training.py
```

### What training does

- loads `data/data.txt`
- builds character, syllable, and phoneme vocabularies
- pads samples to fixed maximum lengths
- initializes the encoder/decoder stack
- resumes from `model/checkpoint.pth` if available
- writes:
  - `model/checkpoint.pth`
  - `model/saved_model.pth`
- updates latent-space statistics during training

### Current training hyperparameters

Defined in `training.py`:

```python
BATCH_SIZE = 16
LEARNING_RATE = 0.001
EPOCHS_TOTAL = 100
HIDDEN_DIM_NO = 256
EMB_DIM_NO = 128
N_LAYERS = 2
DROPOUT = 0.1
```

### Important note about the training loop

The current training script uses:

```python
while True:
```

So training does **not stop automatically** after `EPOCHS_TOTAL`. The scheduler uses `EPOCHS_TOTAL` as the cosine annealing horizon, but the loop itself is open-ended and must be stopped manually.

## Inference

Run interactive inference:

```bash
python inference.py
```

The script opens a prompt:

```text
Enter words/phrases (type exit to quit):
```

Type a word or phrase and the script prints:

```python
(final_transcription, number_of_variants)
```

### Programmatic use

```python
from inference import infer

transcription, n_variants = infer("palavra")
print(transcription)
print(n_variants)
```

### Inference behavior

- input is tokenized on whitespace
- each token is processed independently
- hyphenated tokens are split on `-` and rejoined with `.` in the base representation
- base outputs are wrapped in `/.../`
- `generate_variants()` expands the base output into a final space-separated list of variants

## Output format

### Base output

The neural model produces a **structured archiphonemic transcription**, for example:

```text
/si.ˈla.ba/
```

Properties:

- syllable boundaries are explicit
- stress is explicit
- ambiguous or dialect-sensitive material may remain unresolved in the base representation

### Expanded output

The rule engine returns one or more surface-oriented variants, for example:

```text
/si.ˈla.ba/ /si.ˈla.baɪ/
```

Actual outputs depend on:

- epenthesis opportunities
- archiphonemic markers
- vowel harmony logic
- dialect profile mappings
- rhotic behavior

## Latent space module

`latent_space.py` defines an auxiliary `PhoneticLatentSpace` module with learnable pattern vectors and running statistics such as:

- usage counts
- phoneme co-occurrence counts
- pattern-conditioned context accumulation

The training script updates these statistics once per batch and saves the module state. `inference.py` also exposes an experimental `infer_with_pattern()` function for probing how a selected latent pattern perturbs the output.

## Research scope

This repository is best understood as a **structured pronunciation model** rather than a conventional flat G2P baseline. The system is especially aimed at:

- explicit syllabification,
- lexical stress prediction,
- archiphonemic intermediate representations,
- dialect-sensitive post-neural expansion,
- pseudoword-based evaluation of productive generalization.

## Known implementation notes

- The current scripts are configured for **CPU** execution.
- Maximum lengths are fixed in code:
  - source length: `30`
  - maximum syllable sequence length: `10`
  - maximum phoneme length per syllable: `12`
- The training loop is open-ended and should be stopped manually.
- `infer_with_pattern()` is experimental.
- The latent-space save/load locations may need harmonization depending on how the repository is arranged in a fresh checkout.
